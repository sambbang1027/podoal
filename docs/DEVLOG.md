## 2026-06-14

### E2E 테스트로 드러난 것들: 코드를 실제로 실행해야 보이는 게 있다

#### 배경

Zone/Seat 플로우 개선, floor2 리셋 버그 수정, VIP 난이도 조정까지 마쳤으니
전체 플로우를 자동화 테스트로 한 번에 검증하려 했다. Playwright로 4개 케이스를 작성.

#### 발견 1 — /popup 라우트가 없었다

`CLAUDE.md`에 적힌 플로우: `SETTING → READY → POPUP → QUEUE → ...`  
실제 `App.tsx`: `/popup` 라우트 자체가 없음. 흐름은 `READY → QUEUE`로 바로 이어진다.

`Popup.tsx` 파일은 존재하지만 라우터에 등록되지 않은 데드 코드 상태.  
문서와 코드가 달랐던 것. 테스트를 실제로 실행하지 않았다면 오랫동안 모를 수도 있었다.

#### 발견 2 — Ready 페이지 흐름이 생각보다 복잡하다

처음 테스트 코드는 `00:00.xxx` 텍스트가 나타나기를 기다렸다. 당연히 타임아웃.  
실제 Ready 페이지 흐름:

```
select 단계 → "10초 전" 클릭
    → countdown 단계 (10초 대기)
        → active 단계 → 날짜/시간 선택 패널 표시
            → 예매하기 활성화 → 클릭
```

날짜/시간 선택 패널은 `phase === 'active'`일 때만 DOM에 렌더된다.
카운트다운이 끝나기 전에는 패널 자체가 없다.
`waitForSelector('button:has-text("2026년 06월 06일")')` 로 active 진입을 감지하는 방식으로 해결.

#### 발견 3 — Playwright headless에서 F5는 JS keydown보다 브라우저가 먼저 가져간다

앱의 F5 패널티 구현: `keydown → e.preventDefault() → 오버레이 1.8초`  
headless Chromium에서는 F5가 JavaScript 이벤트 루프보다 브라우저 레벨 새로고침으로 먼저 처리된다.
`e.preventDefault()`가 실행되기 전에 실제 페이지가 새로고침된다.

QUEUE 유지(`page.url().includes('queue')`)는 sessionStorage persist 덕분에 상태 복원된 것이지, 오버레이가 작동한 것이 아니다.

결론: headless 환경에서 F5 인터셉트 동작은 테스트 불가. 실제 브라우저에서 수동 확인으로 대체.
자동화 테스트가 "전부"가 아님을 보여주는 케이스.

#### 오늘의 인사이트

코드 리뷰나 정적 분석으로는 보이지 않는 것들이 있다.  
- 라우트가 실제로 연결되어 있는지  
- 컴포넌트가 어떤 조건에서 DOM에 나타나는지  
- 브라우저 레벨과 JS 레벨의 이벤트 처리 순서  

"실행해봐야 안다"는 말이 추상적으로 느껴지다가, 오늘처럼 구체적인 사례로 만나면 확실히 남는다.

## 2026-06-11

### 실제 티켓팅 시스템 구조 분석 + 대기열 고증 방향 결정

#### 배경
포도알 파이터의 대기열 구현이 실제 멜론티켓 동작과 다르다는 걸 인식.
현재 구조는 F5를 누르면 새로고침이 막히고(preventDefault) 패널티만 숫자로 더해주는 방식인데,
실제 멜론티켓에서는 F5를 누르면 진짜 새로고침이 되고 대기열 번호가 뒤로 밀린다.

#### 실제 티켓팅 시스템이 왜 이렇게 동작하는가

멜론티켓 대기열의 실제 추정 구조:
- 정각에 "예매하기" 클릭 → 서버에 POST 요청
- 서버가 요청 받은 타임스탬프 순서로 Redis에 대기 번호 부여
- WebSocket 또는 SSE로 번호 감소를 실시간 푸시
- F5 누르는 순간 브라우저가 페이지를 처음부터 재로드 → 서버에 새 요청 → 그 시점 기준 새 번호 발급 → 뒤로 밀림

heartbeat(생존 신호 주기 체크) 방식도 생각해봤지만, 이건 과한 구조일 가능성이 높다.
가장 단순하고 유력한 설명은 "재요청 = 새 번호"다.

실제 경험담(직접 겪음): 대기열 번호가 멈춰있어서 F5 눌렀더니 번호가 뒤로 밀렸다.

#### 현재 구현의 문제

```
현재: F5 → preventDefault() → 실제 새로고침 차단 → 패널티 숫자만 추가
실제: F5 → 진짜 새로고침 → 세션 끊김 → 새 번호 발급 → 뒤로 밀림
```

추가로, Zustand 상태가 메모리에만 있어서 새로고침이 실제로 일어나면
상태가 초기화되어 홈으로 튕기는 문제도 있다.

#### 결정: 방향 2 — 새로고침 허용 + sessionStorage persist

- `preventDefault()` 제거 → 실제 새로고침 허용
- sessionStorage + Zustand persist 미들웨어로 상태 유지
- `beforeunload` 이벤트로 새로고침 감지 → 패널티 숫자를 sessionStorage에 저장 후 재진입 시 반영
- 실제 멜론티켓 동작과 가장 유사한 고증

#### 이 프로젝트에서 배울 수 있는 것

단순히 "React로 게임 만들었다"가 아니라:

1. **실제 서비스 역공학** — 멜론티켓의 대기열 동작을 관찰하고 내부 구조를 추론
2. **클라이언트만으로 서버 동작 시뮬레이션** — Redis 대기열, WebSocket 실시간 업데이트, 세션 기반 순번을 백엔드 없이 프론트엔드 로직으로 구현
3. **브라우저 이벤트 심층 활용** — `keydown preventDefault`, `beforeunload`, sessionStorage 영속성
4. **의도적인 설계 결정** — 왜 이렇게 구현했는지 설명할 수 있는 근거가 있다

포트폴리오에서 "왜 이렇게 구현했나요?" 질문이 나왔을 때,
실제 시스템 구조를 이해하고 의도적으로 시뮬레이션했다고 설명할 수 있다.

## 2026-06-12

### VIP 난이도 설계: 이선좌(takenProb)가 핵심이다

#### 문제 인식

VIP를 너무 쉽게 잡을 수 있다. 실제 티켓팅에서 VIP는 거의 항상 실패하는데,
게임에서는 여러 번 시도하면 거의 성공한다. 도파민이 안 터진다.

#### 처음 접근 — 봇 속도를 올리자

봇 수를 늘리고(8→15+), 속도를 올리면(400ms→150ms) VIP가 빨리 소진될 것이라 생각.

그런데 계산해보니:
- 15봇 × (1000 / 150ms) = 100 seats/sec
- 60석 ÷ 100 = 0.6초 — Zone 페이지 로드되자마자 매진

봇을 너무 빠르게 하면 유저가 클릭할 틈도 없이 매진. 그건 어렵게 만드는 게 아니라 막는 것.

#### 핵심 인사이트 — 이선좌(takenProb)

VIP 난이도의 핵심은 "좌석 페이지까지 가는 것"이 아니라 "좌석 페이지에서 성공하는 것".

실제 티켓팅 실패 경험: "눈앞에서 자꾸 사라져요", "클릭했는데 이미 선택됨이 떴어요"  
→ 이게 바로 이선좌(클릭 시 이미 취소된 좌석) 현상.

`takenProb`을 높이면:
- 좌석 페이지에 들어가도 클릭하면 이선좌 모달
- 모달 닫고 다른 좌석 클릭 → 또 이선좌
- 그 사이 좌석은 봇이 계속 소진
- 간신히 성공하거나, 결국 전체 매진

봇 속도보다 이선좌 확률이 "실제 티켓팅" 느낌을 만드는 변수다.

#### 최종 결정

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| botCount | 8 | 10 |
| botMinMs | 400ms | 300ms |
| botMaxMs | 600ms | 500ms |
| **takenProb** | **0.5** | **0.85** |

봇 속도는 적당히 올려 Zone에서 VIP가 빠르게 줄어드는 시각적 긴장감을 주고,
이선좌 0.85로 Seat에서의 성공 자체를 희박하게 만드는 구조.

### 버그 발견: totalTime이 비정상적으로 크게 나오는 문제

#### 원인
```ts
const totalTime = success ? Date.now() - bookingStartTime : 0
```
`bookingStartTime`의 초기값이 `0`이라서,
정상 플로우를 거치지 않으면 `Date.now() - 0 = 현재 유닉스 타임스탬프(ms)` 가 그대로 들어감.
결과: 1781184996초 (약 56년) 같은 말도 안 되는 값 저장.

#### 해결 방향
`bookingStartTime > 0`일 때만 totalTime 계산하도록 조건 추가.
(sessionStorage persist 구현과 함께 적용 예정)
